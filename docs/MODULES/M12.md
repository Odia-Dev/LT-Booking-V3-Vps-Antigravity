# Module M12: Razorpay Payment Integration

Outlines the database models, repository layers, signature checks, webhook parsers, refund architecture, and notifications integration to manage online payments and transactions lifecycle via Razorpay.

---

## Status

🟢 Production Ready (Verified & Build Stable)

---

## Deliverables

- **Database Schema Mappings (`schema.prisma`)**:
  - Created `Payment` table storing transaction refs (`bookingId`, `customerId`, `razorpayOrderId`, `razorpayPaymentId`, amounts, statuses, and raw gateway response payloads).
  - Created `PaymentAudit` log table to persist state transaction audit history.
  - Linked `Payment` to `Booking` (one-to-one) and `User` (one-to-many).
- **Payment Repository (`paymentRepository.ts`)**:
  - Implemented data-access actions: `createPayment()`, `updatePayment()`, `getPaymentById()`, `getPaymentByOrderId()`, `getPaymentByBookingId()`, and `listPayments()`.
  - Configured transaction safety generating initial payment audit log entries upon order creation.
- **Payment Service (`paymentService.ts`)**:
  - Configured Razorpay Client SDK initialization via isolated environment credentials.
  - Developed `createRazorpayOrder()` converting amounts to paise, generating Razorpay orders, and setting the local booking state to `PAYMENT_PENDING`.
  - Implemented `verifySignature()` executing cryptographically secure signature hash validations using `crypto.createHmac` matching Razorpay secret keys.
  - Implemented state updates: `markSuccess()` (moving status to `SUCCESS` and booking to `CONFIRMED`), `markFailed()`, and `markRefunded()`.
- **Zod Schema Input Validation (`paymentValidation.ts` & `refundValidation.ts`)**:
  - Configured Zod schema rules validating order creation payloads (`CreateOrderSchema`) and refund payload limits (`AdminInitiateRefundSchema`).
- **REST Controller Handlers (`paymentController.ts`)**:
  - Exposed controllers for order creation (`createOrder`), detail lookups (`getPaymentById`, `getPaymentByOrderId`), refunds initiation (`initiateRefund`), and history queries (`getRefundHistory`).
  - Added RBAC checks protecting payment data boundaries (Admins get full access; Customers get access only to read their own payments).
- **Routing Configurations (`paymentRoutes.ts`)**:
  - Configured protected administrative endpoints under authentication session checks.
  - Configured a dedicated `publicPaymentsRouter` allowing anonymous guest checkouts to initiate order creation and trigger payment verifications.
  - Mounted routers under `/api/payments` and `/api/public/payments` in `app.ts`.
- **Frontend Razorpay Checkout Flow (`/book-online`)**:
  - Loaded official Razorpay SDK script dynamically.
  - Modified form submission to request backend order IDs and launch the Razorpay Checkout overlay.
  - Configured callback handler posting details to the signature verification endpoint, transitioning screens to **Booking Confirmed** on success, or **Payment Failed** (with a fully functional **Retry Payment** button) on failure/dismiss cancellation.
- **Webhook Processing (`webhookController.ts` & `webhookRoutes.ts`)**:
  - Exposed public webhook receiver `POST /api/webhooks/razorpay`.
  - Configured Express json parser to populate `req.rawBody` buffer.
  - Validates webhook signatures against computed signatures using `RAZORPAY_WEBHOOK_SECRET`.
  - Maps events: `payment.captured` & `order.paid` (marks payment as `SUCCESS` and booking as `CONFIRMED`), `payment.failed`, and `refund.processed`.
  - Enforces duplicate protection (idempotency) by calculating raw body SHA-256 hashes (`payload_hash`) and searching transaction audits before processing events.
- **Refund Architecture**:
  - Added payment transaction statuses: `REFUND_REQUESTED`, `REFUND_PROCESSING`, `REFUNDED`, and `PARTIAL_REFUND`.
  - Created mock gateway dispatch stubs `executeMockRazorpayRefund()` for administrative refunds.
  - Supports tracking partial refunds and maximum refundable limits.
- **Notifications Integration (`bookingNotificationService.ts`)**:
  - Registered `booking.refund_processed` event handler.
  - Added WhatsApp template dispatches across checkout milestones: `booking_confirmed`, `payment_success`, `payment_failed`, and `refund_processed` across Email, SMS, and WhatsApp channels.
  - Connected emitters inside repository/service state transitions.
